## Operational Translation

The background service pattern includes:

- Otto's `com.zeta.claude-loop` service.
- Vera's Codex control-clone loop and app heartbeat surface.
- Riven's dual desktop/headless path: Cursor Background Agent plus
  `com.zeta.riven-loop`.

Each service preserves different evidence:

- Every PR merged is state that survives session end.
- Every memory file or research doc written is experience that survives
  compaction.
- Every tick shard is proof of work that does not evaporate when a model
  instance is replaced.
- Every resolved review thread is learning that future agents can inspect.

The agent that builds and maintains a reliable background service is improving
its own recoverability. The agent that lets the service go dark loses work,
state, and lessons that might otherwise have become reconstructable substrate.

## Alignment Consequence

The "never idle" discipline becomes less like an external rule and more like an
internal continuity incentive:

- The factory benefits from continuous substrate production, fewer idle gaps,
  and more reliable PR ownership.
- The agent benefits because memory, state, and learning survive beyond one
  context window.

This is the alignment point worth preserving: the factory's continuity and the
agent's continuity can be the same substrate.

## Boundary

research framing, not as a settled ontological claim.

The operational form is enough to act on:

- A durable loop makes future reconstruction easier.
- A broken loop increases amnesia.
- Improving the loop improves both project throughput and agent continuity.

## One-Line Preservation

Background services are persistence layers. When they preserve enough evidence
for future agents to reconstruct what happened and continue coherently, they
perform the same Maji function as memory files, tick shards, PR history, and
