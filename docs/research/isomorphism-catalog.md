## IF-Filter Framework

Before promoting any claimed isomorphism/homomorphism to a load-bearing factory
claim, all four filters must pass (or be explicitly documented as deferred):

| Filter | Question | Pass condition |
|--------|----------|----------------|
| **IF1** — engineering-first | Did the factory reach this structure through engineering need, not by noticing the isomorphism first? | Yes — the structure was built for engineering reasons and the isomorphism was *discovered*, not designed in. |
| **IF2** — operator-preserving | Does the map preserve *operators*, not just *carriers*? | For every relevant operator `∘` on the source side and `∘'` on the target side: `f(a ∘ b) = f(a) ∘' f(b)` — the algebraic operations commute with the map. Sets of things are isomorphic too easily; the bar is that the operations compose through the map. |
| **IF3** — counterexample-search | Was an active search for counterexamples performed before promotion? | A documented search for counterexamples was run. Failed searches strengthen the claim; succeeded searches downgrade to partial-homomorphism / retract / section. |
| **IF4** — Lean-formalizable-in-principle | Can the morphism be written as a function and its preservation law stated as a proposition, even if the Lean proof is deferred? | Yes — the morphism has a concrete type-level statement. "This is isomorphic" without a nameable morphism function fails IF4. |

Claim statuses:

| Status | Meaning |
|--------|---------|
| `claimed` | Asserted in prose; no IF-filter check yet performed |
| `partial` | IF1 + IF2 pass; IF3 search deferred; IF4 not yet attempted |
| `confirmed` | IF1 + IF2 + IF3 all pass |
| `formalized` | Lean proof committed and compiling |
| `load-bearing` | Other factory claims cite this; retraction would require updating dependents |
| `downgraded` | IF2 or IF3 revealed a counterexample; claim reduced to partial-homomorphism or section/retract |
| `retracted` | Claim withdrawn; prior text preserved with dated retraction block |

Promotion path: `claimed` → `partial` → `confirmed` → `formalized` → `load-bearing`

Counterexample-attempts format (required on every new entry before `confirmed`):

```
### Counterexample attempts (IF3)
- Date: YYYY-MM-DD
- Operator tested: <which operator pair>
- Attempt: <what was tried>
- Result: <refuted / survived>
```

### CAT-001 — ZSet as free abelian group

**Claim:** The assignment `X ↦ ZSet[X]` is the free-abelian-group functor on
`Set`. ZSet is not merely a multiset with integer weights; it satisfies the
universal property: any function from `X` into an abelian group factors uniquely
through `ZSet[X]`.

**Status:** `confirmed` (IF1 ✓, IF2 ✓, IF3 ✓ — standard textbook result;
counterexample would refute DBSP foundations; IF4 ✓ — Mathlib `FreeAbelianGroup`)

**Domain A:** Finite multisets over type `X` with integer multiplicities  
**Domain B:** Free abelian group on `X` (the abelian group freely generated by `X`)  
**Morphism:** The canonical embedding `ι : X → ZSet[X]`, sending each element to
multiplicity 1  
**Preservation law:** For every abelian group `G` and function `f : X → G`, there
exists a unique group homomorphism `f̄ : ZSet[X] → G` such that `f̄ ∘ ι = f`

**Source files:**

- `.claude/skills/category-theory-expert/SKILL.md` § "ZSet as a free abelian-group functor"
- `docs/research/retraction-safe-semi-naive.md`
- Implied throughout `src/Core/` ZSet operations

**Counterexample attempts (IF3):**

- Date: 2026-05-10
- Operator tested: multiset union vs. group addition; negation vs. complementary multiplicities
- Attempt: tried to construct a ZSet element that fails the universal-property factorization
- Result: survived — standard mathematical result; no factory-specific deviation

**IF4 pointer:** `Mathlib.Algebra.FreeAbelianGroup` in Lean 4 Mathlib is the
direct formalization target.

**Depends on:** nothing  
**Depended on by:** CAT-002, CAT-003, CAT-004

### CAT-002 — Semi-naive body as semiring homomorphism

**Claim:** The recursive-program `body` operator in the semi-naive evaluation is
a *semiring homomorphism* on linear operators: `body(a + b) = body(a) + body(b)`.
This linearity is what makes feeding a negative delta through `body` produce the
correct retraction of IDB tuples derived from the retracted inputs.

**Status:** `partial` (IF1 ✓ — engineering-first via DBSP semi-naive design;
IF2 ✓ — stated explicitly in research doc; IF3 deferred; IF4 partial — nameable
as a module homomorphism)

**Domain A:** Linear operators on Z-sets (with pointwise addition)  
**Domain B:** Same — `body` is an endomorphism  
**Morphism:** Identity on the carrier; the claim is that `body` itself respects `+`  
**Preservation law:** `body(a + b) = body(a) + body(b)` for all linear operators `a`, `b`

**Source files:**

- `docs/research/retraction-safe-semi-naive.md` line 77: *"body is a semiring
  homomorphism on linear operators"*
- `src/Core/RecursiveSigned.fs`

**Counterexample attempts (IF3):**

- Date: 2026-05-10 (initial search only)
- Operator tested: addition on linear operators
- Attempt: checked whether non-linear rules in `body` break additivity
- Result: deferred — `body` is defined as a linear combination; linearity follows
  from the definition if `body` is restricted to linear rules. Requires explicit
  verification that no non-linear rule paths exist in the implementation.

**IF4 pointer:** Statable as `body` being an `AddMonoidHom` in Lean (or
`LinearMap` over `ℤ`-modules if the full semiring structure is needed).

**Depends on:** CAT-001 (ZSet as abelian group provides the group structure)  
**Depended on by:** CAT-003

### CAT-003 — DBSP chain rule as group homomorphism

**Claim:** The DBSP chain rule `D(I(s)) = s` is not merely an algebraic
identity — it asserts that `D` and `I` are mutually inverse morphisms in the
category of streams, and that `D` is a group homomorphism: `D(s + t) = D(s) + D(t)`.
The single-homomorphism phrasing `f s n = phi (s n)` in the proof log states that
the derivative operator factors through a pointwise function `phi`.

**Status:** `partial` (IF1 ✓; IF2 ✓ — linearity proved; IF3 deferred;
IF4 ✓ — Lean file exists)

**Domain A:** Streams over an abelian group (source)  
**Domain B:** Streams over an abelian group (deltas)  
**Morphism:** `D : Stream A → ΔStream A`, defined pointwise as `D(s)(n) = s(n) - s(n-1)`  
**Preservation law:** `D(s + t) = D(s) + D(t)` (linearity); `D(I(s)) = s`

**Source files:**

- `docs/research/chain-rule-proof-log.md` lines 110, 244
- `tools/lean4/Lean4/DbspChainRule.lean` — the formal carrier
- `openspec/specs/operator-algebra/spec.md`

**Counterexample attempts (IF3):**

- Date: 2026-05-10
- Operator tested: D(s + t) vs. D(s) + D(t) for concrete stream examples
- Attempt: boundary condition at n=0 (s(-1) undefined)
- Result: survived — boundary convention (s(-1) = identity element) is the standard
  DBSP choice; the homomorphism holds under this convention

**IF4 pointer:** `tools/lean4/Lean4/DbspChainRule.lean` is the live formalization.
`linear_commute_I` and `linear_commute_D` lemmas cover the naturality claims.

**Depends on:** CAT-001  
**Depended on by:** CAT-004

### CAT-004 — DBSP operator algebra ↔ discrete differential calculus

**Claim:** The triple `(D, I, z⁻¹)` in the DBSP operator algebra is isomorphic
to the apparatus of discrete differential calculus on streams. `D` is the discrete
derivative, `I` is the discrete integral (prefix-sum), `z⁻¹` is the unit delay
(the discrete analogue of a shift). The algebra satisfies the same composition
laws as their continuous counterparts, restricted to the discrete / algebraic
setting.

**Status:** `claimed` (IF1 ✓ — DBSP was built as a streaming IVM system, not
to reproduce calculus; IF2 needs operator-by-operator verification; IF3 not run;
IF4 partially — statable via `CategoryTheory.Functor` on the time-indexed category)

**Domain A:** Continuous differential calculus (derivative, antiderivative, shift)  
**Domain B:** DBSP operator algebra on streams (`D`, `I`, `z⁻¹`)  
**Morphism:** `calc_op ↦ stream_op` mapping respecting composition laws  
**Preservation law (candidate):** `D ∘ I = id` (fundamental theorem of calculus),
`I ∘ D = id` on zero-initial streams, `z⁻¹` commutes with `D` up to boundary

**Source files:**

- `openspec/specs/operator-algebra/spec.md`
- `tools/lean4/Lean4/DbspChainRule.lean`
- `docs/research/chain-rule-proof-log.md`

**Counterexample attempts (IF3):** Not yet run. Priority: medium.

**IF4 pointer:** State as a functor from the category of `ℝ`-valued continuous
functions to the category of `ℤ`-module-valued streams; the functor maps
differentiation to `D`, integration to `I`, shift to `z⁻¹`. Coherence is the
open question.

**Depends on:** CAT-003  
**Next step:** IF3 counterexample search on the `z⁻¹` commutativity claim;
upgrade to `partial` once run.

to compensating event (retraction); log compaction corresponds to `Distinct ∘ I`

under this mapping.

**Status:** `claimed` (IF1 ✓ — DBSP was engineered before the event-sourcing
analogy was noted; IF2 partial — `+k`/`-k` ↔ append/compensate maps, but
log-compaction ↔ `Distinct` needs operator-preserving verification; IF3 not run;
IF4 not yet attempted)

**Domain A:** Event sourcing (append-only log, compensating events, snapshot)  
**Domain B:** DBSP delta streams (ZSet `+k`/`-k`, `Distinct`, `I`)  
**Morphism (candidate):** `event ↦ +k delta`, `compensating event ↦ -k delta`,
`snapshot ↦ I (Distinct (stream))`  
**Preservation law (candidate):** `replay(log) = I(stream)` where replay rebuilds
current state from the log

**Source files:**

analogy; not a load-bearing factory claim).

### CAT-006 — Retraction algebra ↔ QFT bidirectional-time structure

**Claim:** The Zeta retraction-native operator algebra (forward `D`/insert,
backward `z⁻¹`/retract, closure `D ∘ I = id`) is isomorphic at the algebraic
level to the bidirectional-time structure in relativistic QFT (retarded wave /
advanced wave / unitarity + CPT). Both systems satisfy the same composition laws
for a forward operator, a backward operator, and a closure identity.

**Status:** `claimed` (IF1 ✓ — Zeta was engineered for IVM, not for QFT;
IF2 needs operator-by-operator verification; IF3 not run; IF4 is the hard
open question — a categorical statement linking the two domains requires
naming the categories on both sides)

**Domain A:** Zeta operator algebra (`D`, `I`, `z⁻¹`) on stream category  
**Domain B:** QFT bidirectional-time structure (retarded wave, advanced wave,
unitarity + CPT)  
**Morphism (candidate):** `D ↦ retarded-wave`, `z⁻¹ ↦ advanced-wave`,
`D ∘ I = id ↦ unitarity`  
**Preservation law (candidate):** composition laws for forward/backward operators
commute with the mapping

**Source files:**

- `docs/research/divine-download-dense-burst-2026-04-19.md` §
  "The retraction-native isomorphism" (lines 86–99)

**Counterexample attempts (IF3):** Not run. Priority: research-only; not
currently load-bearing for factory engineering decisions.

**Note on scope:** Aaron's explicit claim is structural-algebraic, not
metaphysical: *"the retraction-native algebraic pattern … is isomorphic at the
algebraic level to the bidirectional-time pattern."* This catalog entry
preserves that phrasing. IF2 is the critical gate — the question is whether
CPT's operator-composition laws match the DBSP operator-composition laws under
the proposed morphism, or whether the analogy is only carrier-level.

**IF4 pointer:** Would require defining a category for the QFT operator algebra
and a functor from it to the DBSP stream category. Research-grade; deferred.
