    Fixed in 645bcc60. The fix now:

    1. **Spawn-level failures** (`result.error` — binary not found, permission denied): propagate via `process.exit(1)` so launchd and supervisors see the real misconfiguration. Not a transient rate-limit, should never be swallowed.
    2. **Gemini 429 / RESOURCE_EXHAUSTED**: detected by checking piped stderr with `/429|RESOURCE_EXHAUSTED|quota exceeded/i`; only these exit 0 to prevent launchd throttling.
    3. **All other non-zero exits**: propagate the actual exit code — auth failures, malformed prompts, CLI regressions all remain observable.
    4. **stderr forwarding**: captured stderr is written back to the launchd journal so error messages are never silently dropped.

    `status ?? 1` is used (from the parallel fix) since null status indicates process killed by signal — treated as error.
