Captures the procedure pattern that emerged from this session's 5 ID-collision-resolution PRs ([#3053](https://github.com/Lucent-Financial-Group/Zeta/pull/3053) B-0444, [#3057](https://github.com/Lucent-Financial-Group/Zeta/pull/3057) B-0068.1, [#3058](https://github.com/Lucent-Financial-Group/Zeta/pull/3058) B-0090.x batch, [#3065](https://github.com/Lucent-Financial-Group/Zeta/pull/3065) B-0370-0373 batch, plus the in-flight audit tool [#3056](https://github.com/Lucent-Financial-Group/Zeta/pull/3056)).

## Key precedence rule documented

**External-references trumps first-merged-wins.** The row that would cost more to migrate keeps its ID, regardless of who filed first. Both [#3057](https://github.com/Lucent-Financial-Group/Zeta/pull/3057) and [#3065](https://github.com/Lucent-Financial-Group/Zeta/pull/3065) bent first-merged-wins this way after re-examining external refs.

## 9-step recipe

1. Identify colliding rows via `audit-duplicate-row-ids.ts`
2. Find external references (parent body, sibling `depends_on`, memory, PR history, BACKLOG.md)
3. Apply precedence rules in order (external-refs > first-merged > status-precedence)
4. Pick next-free IDs (check `origin/main` AND open PRs)
5. Create isolated worktree (multi-Otto split-brain prevention)
6. Rename files + edit frontmatter (`renumbered_from:` + reason)
7. Batch connected components (preserve internal dep chain)
8. Update parent's body + regen BACKLOG.md
9. Verify + commit + push + PR

- Initial analysis can point wrong direction (re-examine when rules disagree; PR #3065 reversed first read after re-checking timeline)
- ID collisions hide in substrate for weeks (run audit BEFORE filing, not AFTER reviewers catch)
