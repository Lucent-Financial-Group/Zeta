## Description

Extends sub-case 5 of [\`.claude/rules/holding-without-named-dependency-is-standing-by-failure.md\`](.claude/rules/holding-without-named-dependency-is-standing-by-failure.md) with a push-time mitigation pattern empirically validated this session.

## The pattern

When peer Otto's \`git switch\` race moves HEAD between commit and push, the standard \`git push -u origin <branch>\` reads CURRENT HEAD at push time (which may have moved off your branch) and pushes the wrong branch to your intended remote ref. The safer pattern:

\`\`\`bash
git branch <new-branch-name> <commit-sha>
git push origin <new-branch-name>:<new-branch-name>
\`\`\`

This bypasses HEAD-state confusion because:

1. \`git branch <name> <sha>\` creates the ref without changing HEAD (peer Otto's HEAD movements don't affect the named ref)
2. \`git push origin <src>:<dst>\` reads the source ref directly, not HEAD
3. The commit-SHA is the durable handle — even if local branches get poisoned with peer's commits, the SHA still resolves to your work

## Empirical anchor

Session 2026-05-16T16:30Z-16:40Z had ≥3 mid-commit HEAD-contamination events while attempting to push a rule edit. The first 2 push attempts (\`git switch -c\` + \`git push -u\`) failed by pushing the wrong content to the named remote ref. PR [#3910](https://github.com/Lucent-Financial-Group/Zeta/pull/3910)'s successful merge used the explicit-branch-push sequence on the third attempt.

## Self-applying

This PR itself uses the pattern: commit landed on \`otto-cli-explicit-push-mitigation-...-1712z\`; the branch ref pushed via \`git branch otto-cli-explicit-push-applied-...-1713z <sha>\` + \`git push origin <name>:<name>\`. No HEAD movement during push step.
