## Summary

- Adds 36 `permissions.allow` entries to `.claude/settings.json` mapping the substrate-side don't-ask-permission authority model onto harness-side permission rules
- Covers factory tool patterns: `bun` (TS runner), `dotnet` (build/test), `gh` (GitHub CLI), git write operations, `z3`, `Edit`, `Write`, `WebFetch`, `WebSearch`
- Decomposes B-0160 from atomic → parent with 3 slices; this PR is slice 1

### Design choices

- **Granular git patterns** (not `Bash(git *)`) — keeps destructive operations (`reset --hard`, `clean -f`) behind the prompt gate
- **Granular gh patterns** — allows `gh pr/issue/api/run/label` but NOT `gh repo delete`
- **`Edit` + `Write` + `WebFetch` + `WebSearch` fully allowed** per the broad authority model (don't-ask-permission rule)

### What this does NOT cover (future slices)

- Hardcoded safety gates vs settings-allowable actions (Tick-6 merge-denial investigation)
- CLAUDE.md documentation of the integration pattern

### Research findings (Otto-364 search-first)

Per [code.claude.com/docs/en/permissions](https://code.claude.com/docs/en/permissions) (searched 2026-05-08):
- Rules evaluate **deny → ask → allow** (first match wins)
- Settings precedence: managed > CLI > local project > shared project > user
- Bash patterns support glob `*`; space before `*` enforces word boundary
- Compound commands split by `&&`, `||`, `;`, `|` — each subcommand matched independently
