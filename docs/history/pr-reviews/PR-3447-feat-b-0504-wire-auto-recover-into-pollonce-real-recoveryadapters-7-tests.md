```
fix(b-0504): detach HEAD before `branch -D` in gitCreateBranch retry

Codex PR #3447 line 344: `git branch -D <recovery/N>` CANNOT delete
the currently-checked-out branch. If a prior recovery attempt failed
mid-flight and left HEAD on `recovery/<prNumber>`, the delete fails
with "cannot delete branch ... used by worktree", AND the subsequent
`git checkout -b` also fails because the branch already exists.
Result: retries wedge permanently, violating the B-0503 retry-safety
contract this adapter is meant to implement.

Mitigation: `git checkout --detach <base>` BEFORE `git branch -D
<branch>`. Detaching HEAD off the recovery branch means `branch -D`
succeeds. The intermediate detached state is transient — we
`git checkout -b <branch> <base>` immediately after, so there's no
window where the working tree is in a meaningfully bad state.

This makes `gitCreateBranch` truly retry-safe across all the
realistic prior-failure states (clean HEAD, HEAD on main, HEAD on
the recovery branch from a failed attempt). The pre-mutation
working-tree gate from the previous commit (`isWorkingTreeClean`)
remains the operational guard against dirty working trees;
detach-then-delete handles the clean-but-on-the-wrong-branch case.

Test coverage: the test stub adapters in detector.test.ts don't
exercise this real-adapter path (they inject fake behavior); the
fix is operational in REAL_RECOVERY_ADAPTERS. End-to-end retry
exercise lives in B-0505's eventual integration tests.

Co-Authored-By: Claude <noreply@anthropic.com>
```
