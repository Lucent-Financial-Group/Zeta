## Substrate-honest observations

### Self-shard-PR off-by-one count error class

When writing a tick shard's carry-forward "in-flight PRs" count, the shard's OWN PR is in-flight at write-time (PR is opened before the shard is committed to its branch). The 0947Z shard said "1 in-flight (#3400)" but should have said "2 in-flight (#3400, and this shard's PR which will be #3401)".

Same error in 0940Z's carry-forward (claimed "6 in-flight" but only 5 enumerated; the 6th would have been the shard's own PR which gets #3400).

**Mitigation for future-Otto**: when listing in-flight PRs in a shard:

- Always include "this tick's shard PR" as one of the in-flight PRs
- OR phrase the count as "N enumerated + this tick's PR = N+1 in-flight"

The forward-prediction-by-1 logic also applies for the count, not just the PR number — and as 0813Z's PR-number-forward-prediction failure showed, forward-prediction is unreliable under concurrent peer-Otto activity. Better: phrase the count as "N enumerated + this tick's PR" to avoid the prediction.

### Substrate-honest acknowledgment of merged errors

Both off-by-one errors are on `origin/main` in the 0947Z + 0940Z shards. Can't fix retroactively without separate corrective PR. The substrate-honest acknowledgment is the reply on the closed PR thread + recording the lesson here.

If a future canonicalization PR covers count discipline, it should cite empirical evidence (these two off-by-one occurrences) + the mitigation pattern.
