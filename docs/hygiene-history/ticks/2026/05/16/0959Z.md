## Multi-Otto contamination — empirical anchor for future-Otto

The 0944Z tick exposed a previously-undocumented failure mode:
when peer Otto-CLI fires every minute on the same physical
worktree, `git switch -c shard/tick-<HHMM>z-otto-cli-<date>
origin/main` is RACY because peer's identical naming convention
will collide within the ~5-second commit window.

**Mitigation**: topic-namespaced branch names that peer's
shard-tick auto-generator won't reproduce. The recovery branch
`otto-cli-b0418-audit-recover-2026-05-16-0944z` worked. Future
tick shards from a contention-prone session may want:

- `otto-cli-<topic>-<date>-<HHMMZ>` for substantive tick shards
- Reserve `shard/tick-<HHMM>z-otto-cli-<date>` for the lower-
  contention case (single-Otto sessions)

The reflog SAVED this tick: even after peer's commit
superseded mine on the shared-name branch, my dangling commit
remained findable by SHA, and a cherry-pick onto a uniquely-
named branch fully recovered the work.

Worth filing this as a substrate-honest backlog row if not
already covered by B-0519 (multi-Otto contamination RCA). The
shard-name-collision sub-class extends B-0519's enumeration.
