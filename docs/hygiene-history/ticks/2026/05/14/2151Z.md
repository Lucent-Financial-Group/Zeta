## Temporal-provenance discipline codified

Future minimal-continuation shards should choose ONE of three patterns:

1. **Stack on prior shard's PR**: use `gh pr create --base <prior-branch>`
   so the tree at review-time contains the cited prior shard.
   Trade-off: PRs land as a chain, can't auto-merge independently.

2. **Inline the minimal audit-sweep context**: make the shard
   self-contained for readers seeing only this PR's diff.
   Trade-off: small content duplication; defeats the "minimal"
   intent.

3. **Reply at review-time + resolve**: explain the temporal context
   (the prior shard's PR is wait-ci with autoMerge armed; the cite
   will be navigable by merge-time). Trade-off: requires the reviewer
   to trust the resolution explanation.

For this session's pattern (cron firing every minute; shards merge in
order over a few minutes), option 3 is pragmatic. For longer-windowed
PRs, option 1 or 2 is better.
