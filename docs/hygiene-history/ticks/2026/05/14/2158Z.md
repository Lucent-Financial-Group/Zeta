Inspected `tools/hygiene/audit-md032-plus-linestart.ts` as the

Initial scope idea: detect `[text](.claude/...)` or
`[text](docs/...)` markdown links inside tick-shard files (the bug
class Copilot caught on #3264 → #3269).

**Pre-implementation discovery**: a naive regex
(`\]\(\.claude/|\]\(docs/`) hits many false positives in EXISTING
tick shards because the same pattern appears in:

- Prose describing the bug (e.g., `` `](.claude/` `` patterns)
- Inline code spans (`` `[link](docs/foo.md)` ``)
- Fenced code blocks containing example URLs

A correct implementation needs to:

1. Skip content inside inline code spans (single + double backtick)
2. Skip content inside fenced code blocks
3. Match only actual markdown link syntax `[text](path)` in prose

CommonMark parsing is non-trivial. The right implementation either:

- Uses a CommonMark AST library (new dependency for this single audit)
- Implements a small state machine tracking code-span / fence state
- OR is intentionally scoped narrower (e.g., only flag the pattern in
  files NOT containing `bash`/`grep` mentions — too brittle)

Estimated effort: **M (medium)**, not single-tick **XS**. Multi-tick
work spanning design + implementation + tests + false-positive triage.

**Decision**: defer to a future tick (or multi-tick effort) where
this can be implemented carefully rather than rushed. The
audit-discipline-candidate filing list (in 2146Z + 2154Z) remains
the substrate-honest record of the work; a real backlog row is the
right next step if it persists.
