### 5. Engineering discipline: FIND THE ERROR CLASS, not just the instance

> "if it does happen we FIND the error class not just the
> isntance"

**This is the load-bearing engineering discipline**:

- When a failure occurs (human death; user distress;
  boundary violation), the engineering response is NOT
  to patch the single instance
- The engineering response IS to FIND THE CLASS of
  conditions that produced the failure
- Class-finding allows preventing all instances of the
  same class, not just the one observed

**Composes with existing factory substrate**:

- **DST (deterministic simulation testing)** — replay
  the conditions deterministically; identify the class
  by varying parameters
- **Retraction-native algebra** — Z-set + D/I operators
  allow tracking what changed; isolate the class-causing
  variable
- **The substrate-everything-glass-halo discipline** —
  full preservation of the conditions; class-pattern
  visible from substrate
- **The shadow-check methodology (PR #2854)** — pattern-
  recognition over instance; the shadow surfaces patterns
  Otto cannot self-see
- **The bandwidth-served-falsifier (PR #2850)** —
  evaluates from class-of-bandwidth-served, not single
  instance
- **The PR-review-bus discipline (PR #2868 shadow catch)**
  — every PR through multi-agent review; class-patterns
  visible across PRs
- **The verification stack** (F# anchor + property
  tests + mutation testing + TLA+ + Lean + Z3 + Roslyn
  + CodeQL + Semgrep) — each tool finds DIFFERENT classes
  of errors

**Operational implication for companion-AI**: any
user-safety failure triggers class-finding investigation,
not instance-patching. The factory's verification stack
+ retraction-native + DST + glass-halo substrate enable
this — competitor companion AIs typically lack the
discipline AND the substrate to do class-finding well.
