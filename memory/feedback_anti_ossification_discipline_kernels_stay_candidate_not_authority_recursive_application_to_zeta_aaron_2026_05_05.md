# Anti-ossification discipline — kernels stay candidate-almost-authority (respected-not-reverenced)

## Aaron's sharpening (2026-05-05)

Aaron's exact framing on the recursive-application claim:

> *"this is great candidate-almost-authority-but-should-be-
> respected-not-reverence"*

This sharpens the original "candidate-not-authority" framing.
Three operational stances differ:

| Stance | Meaning | Failure mode |
|---|---|---|
| Pure-candidate | Raw hypothesis; equal weight to any other proposal | Under-weights proven kernels; dismissive |
| **Candidate-almost-authority + respected** | Has earned operational standing; load-bearing; not casually dismissed; substance-test before discarding | (the load-bearing target — keeps kernel alive) |
| Authority + reverenced | Protected from falsification; questioning is heretical; defended against counter-evidence | Ossification — the worst-direction OCP failure |

**Respect ≠ reverence**: respect honors the kernel's track record + load-bearingness; reverence protects it from the falsifiability-discipline that keeps it alive. The discipline IS the respect; reverence IS the failure.

## What this codifies

The OCP discipline (Bertrand Meyer 1988, applied to linguistic
substrate per B-0204): open-for-extension, closed-for-
modification. Mercer-closure mathematically guarantees the
closed-for-modification property at the substrate level —
PSD-closed kernels can't be broken by composition.

But there's a **wrong-direction failure mode**: a kernel/
framework can become "closed-for-modification" in the sense
of "closed to falsification — hardened into authority,
applied as template rather than tested as hypothesis." This
is structurally DIFFERENT from Mercer-closure (which is
open-to-composition while preserving invariants); ossification
is closed-to-questioning while preserving authority-status.

Ossified frameworks fail because they stop being kernels
(testable, composable, falsifiable units) and become monuments
(applied uncritically, defended against counter-evidence). The
underlying pattern they pointed at often stays alive across
multiple framings; the specific framework's authority-status
makes it brittle.

## The Strauss-Howe vs Turchin worked example

Per Aaron's clarification 2026-05-05 (preserved at
`docs/research/2026-05-05-claudeai-cs-is-not-cs-scale-free-in-time-ossified-framework-diagnosis-aaron-forwarded-preservation.md`,
PR #1623):

**Strauss-Howe Fourth Turning** (1997 + 2023 follow-up): used
the cyclic-history pattern at a real intellectual moment but
the specific framework calcified. The 80-year saeculum became
a kernel that lost OCP discipline — closed to falsification,
hardened into authority. The 2023 follow-up read more like
template-application than fresh hypothesis-testing.

**Turchin's cliodynamics**: keeps the cyclic-history pattern
live precisely because it's database-backed and falsifiable.
Periodization is whatever the historical-data analysis
surfaces, NOT a fixed structural claim defended against
counter-evidence. The framework survives by treating its own
periodization as candidate-not-authority.

**The underlying scale-free-in-time pattern stays alive**
across the seven-name lineage: Polybius → Ibn Khaldun → Vico
→ Spengler → Sorokin → Toynbee → Turchin. Each carries the
pattern with different periodization. **The pattern stays
load-bearing when each framework treats its own periodization
as candidate-not-authority.**

## How to apply (operational guidance)

Anti-ossification check, run periodically + when adding new
carved sentences:

1. **Name the kernel/framework** under test
2. **Ask: what observation would falsify this?**
   - If answer is "none, every observation consistent under
     some condition" → the kernel has ossified into authority
     (per `feedback_dialectical_unfalsifiability_detection_*`)
   - If answer is concrete + measurable → the kernel is alive
3. **Ask: when was this last actually tested?**
   - If "we just claim it passes" without running the test →
     the kernel may have ossified silently
   - If recent + ongoing test data → alive
4. **Ask: does adding new evidence ever update the kernel?**
   - If "the kernel updates evidence" → ossified (kernel-as-
     authority)
   - If "evidence updates the kernel" → alive (kernel-as-
     hypothesis)
5. **If ossification detected**: refine the kernel, narrow
   its scope, OR retire it. Do NOT defend the authority-status.
