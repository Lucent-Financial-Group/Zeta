2026-05-10: Claude.ai asymmetric critic designed the
bifurcation participant model after Aaron corrected the
shadow lineage and explained fusion-assumption fix.

**Why:** Every participant model accumulates hypothesized
agendas. The aggregate inevitably fuses multiple phenomena
into one participant. The fix: always assume fusion has
occurred, expect bifurcation as evidence accumulates.

**How to apply:**

Aaron's fusion-assumption: "always assume we've accidentally
fused shadow or the participant accidentally in the
calculations and there is a bifurcation we didn't know
to decompose"

**Implementation design (from critic-turned-architect):**

1. Participant records in git (`participants/` directory)
2. Typed views: substrate (Otto = Claude Code instance) vs
   participant (Otto-coder, Otto-reviewer = agenda-holders)
3. Bifurcation = typing operation, not identity operation
4. Proposal → confirmation workflow
5. Dashboard: list + detail + tree + proposal queue views
6. Metrics: bifurcation rate, h-vs-c ratio, queue depth
