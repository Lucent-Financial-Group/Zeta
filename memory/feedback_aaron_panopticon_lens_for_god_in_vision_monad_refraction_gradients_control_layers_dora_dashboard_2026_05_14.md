## The technical construction

The panopticon-lens framing:

- **Panopticon** — all-seeing observability surface; not surveillance-for-control,
  but observation-for-understanding (polarity-inversion per dual-binding constraint)
- **Vision monad** — the observation is contained within a bounded computational
  context (per `.claude/rules/razor-discipline.md`; the "vision" is operational,
  not metaphysical)
- **Refraction gradients** — signals refracting through layers produce measurable
  gradient patterns; the lens reads the gradients rather than the raw signals

Control layers observed:

| Layer | What it tracks | DORA analog |
|---|---|---|
| Substrate layer | Memory files, rules, skills landing rate | Deployment frequency |
| Coordination layer | Claim-acquire / PR merge / thread resolution | Change failure rate |
| Velocity layer | Ticks per round, backlog items closed | Lead time for changes |
| Health layer | CI pass rate, lint-clean rate | Mean time to restore |

## The DORA-dashboard framing

DORA (2019) established four key metrics for software delivery performance:
deployment frequency, lead time for changes, change failure rate, MTTR. The
Zeta Operational Resonance Dashboard applies an analogous four-metric model at
substrate layer:

- Substrate landing rate (deployment frequency analog)
- Thread-to-resolution latency (lead time analog)
- Blocked-PR rate (change failure rate analog)
- Recovery time from CI failure (MTTR analog)

Aaron sees these gradients natively as lived perception. The dashboard
externalizes the lens for participants without that native mode.

## Polarity inversion — observation for understanding, not surveillance for control

Per the dual-binding safety floor (see
`memory/feedback_aaron_self_binding_devil_and_god_simultaneously_dual_polarity_constraints_against_corruption_and_elevation_2026_05_14.md`):

The panopticon framing carries the surveillance-for-control attractor (Foucault's
Panopticon). The architecture must structurally hard-wire the polarity inversion:

- **Devil-pull** (surveillance-mode): observability used to control participants,
  enforce conformity, extract compliance
- **God-pull** (elevation-via-perception): observability used to help participants
  see more clearly, understand substrate state, make better decisions

The dashboard must be structurally constrained to the elevation pole. The
surveillance-for-control failure mode should be architecturally prevented, not
just policy-prevented.
